Public Build System Plan
Product Goal

Create a public reusable build framework for Adobe CEP / hybrid extensions that standardizes:

validation

drift detection

packaging orchestration

installer assembly

release bundling

CI integration

Not a replacement for Adobe signing or Apple tooling.
It is a framework around them.

1. Product Shape
Proposed Name

LBE Build Kit
or
Extension Release Engine

Product Model

Two parts:

A. Core Engine

Reusable for any extension

B. Project Config

Specific to each extension

2. What It Should Support
Core Commands
ext-build check
ext-build verify
ext-build diff
ext-build simulate
ext-build release
ext-build doctor
ext-build init
Command Purpose
check

runs gates only

no packaging

validates config + paths + version sync

verify

checks build output integrity

confirms required files exist

confirms no forbidden files ship

diff

compares current build vs previous artifact

detects removed / added / changed files

simulate

performs full local release build

produces all artifacts locally

no publish

release

full production orchestration

only for CI or authorized local run

doctor

checks environment

missing secrets

missing tools

wrong OS-specific dependencies

init

scaffolds a new extension project config

3. Framework Architecture
Layer 1 — Core Engine

Reusable logic only

packages/
  core/
    cli/
    config/
    pipeline/
    validators/
    diff/
    artifacts/
    logging/

Responsibilities:

parse config

run steps in order

fail-fast

print structured results

support local + CI modes

Layer 2 — Adapters

Tool-specific wrappers

packages/
  adapters/
    zxp/
    inno/
    electron-dmg/
    apple-notary/
    checksum/
    github-release/

Responsibilities:

call external tools

normalize outputs

surface errors consistently

Layer 3 — Extension Profile

Per-project declarative config

ext-build.config.json

Example:

{
  "name": "Letterblack GenAI",
  "extensionId": "com.letterblack.genai",
  "versionFiles": [
    "package.json",
    "src/CSXS/version.json",
    "tools/config/identity.config.json"
  ],
  "distPath": "dist",
  "manifestPath": "src/CSXS/manifest.xml",
  "requiredFiles": [
    "index.html",
    "CSXS/manifest.xml",
    "jsx/LB_DocablePanel.jsx"
  ],
  "forbiddenPatterns": [
    ".env",
    "*.map",
    "manifest.dev.xml",
    "**/tests/**"
  ],
  "artifacts": {
    "zxp": true,
    "windowsExe": true,
    "macDmg": true
  }
}
Layer 4 — Project Plugins

Feature-specific checks

plugins/
  gate-runtime-boundary.mjs
  gate-host-range.mjs
  gate-scriptui.mjs
  gate-license-flow.mjs

This is how public users extend it without editing core.

5. Use Your Current Extension as the Reference Model

Your current extension should become the reference implementation, not the framework itself.

Extract from current extension:

drift gates

artifact completeness checks

version consistency checks

release integrity logic

installer policy checks

runtime boundary validation concepts

secret-surface rules

dual-scope install checks

Split Features into Two Groups
A. Generic Features

Reusable publicly

version sync

artifact completeness

forbidden file scan

manifest consistency

release integrity

checksum creation

bundle structure validation

CI preflight

B. Product-Specific Features

Stay extension-specific

provider logic

Gumroad flow

AE-specific runtime behavior

LBE business logic

license UI behavior

model registry

proprietary execution paths

6. Best Public Design Principle
Do not expose your app internals as the framework

Instead:

Current extension = reference project
Framework = extracted infrastructure layer

That keeps:

public package clean

proprietary logic private

framework reusable

Development use cases

ext-build check → validate while coding

ext-build watch → detect drift during development

ext-build doctor → environment/tool check

ext-build dev-verify → confirm dev build is still structurally valid

ext-build simulate → pre-release local test

Best model
dev mode → fast checks
release mode → strict full pipeline
